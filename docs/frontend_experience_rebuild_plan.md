# 前端体验改造整体方案

更新时间：`2026-03-24`

## 1. 文档目的

这份文档用于沉淀当前仓库 Web 前端的整体改造方案，目标不是单纯“美化页面”，而是把现有控制台式前端升级为一个面向 AI 任务执行、治理控制、可恢复运行和多 Agent 扩展的产品化工作台。

本文档基于当前仓库中的真实实现整理，重点参考以下文件：

- [`apps/web/index.html`](/opt/ai-assistant/apps/web/index.html)
- [`apps/web/assets/dashboard.js`](/opt/ai-assistant/apps/web/assets/dashboard.js)
- [`apps/web/assets/dashboard.css`](/opt/ai-assistant/apps/web/assets/dashboard.css)
- [`README.md`](/opt/ai-assistant/README.md)
- [`docs/runbook.md`](/opt/ai-assistant/docs/runbook.md)

## 2. 当前前端现状

### 2.1 当前实现事实

当前 Web 前端是一个原生静态控制台，主要由单个 HTML、单个 JS 和单个 CSS 文件构成：

- 页面结构主要集中在 [`apps/web/index.html`](/opt/ai-assistant/apps/web/index.html)
- 页面交互、状态管理、API 调用主要集中在 [`apps/web/assets/dashboard.js`](/opt/ai-assistant/apps/web/assets/dashboard.js)
- 视觉样式主要集中在 [`apps/web/assets/dashboard.css`](/opt/ai-assistant/apps/web/assets/dashboard.css)

当前页面已经覆盖了以下能力入口：

- 任务输入、输入分流、草稿确认、fast path
- 任务列表与任务详情
- 步骤、traces、审批、agent runs、session 详情
- session 浏览器
- 治理区与监控区
- actor / 模型 / 工具 / 风险策略 / 变更单相关控制面
- 长期记忆检索

### 2.2 当前前端的主要优点

- 功能覆盖面广，主干能力已经能在 Web 端完成演示与操作
- 治理入口完整，适合当前平台型阶段的联调、验证和故障恢复
- 没有引入复杂前端构建体系，部署简单，联调成本低
- 与当前 API 主链契合度高，适合继续快速验证新能力

### 2.3 当前前端的主要问题

- 信息密度过高，首页同时承担了任务输入、任务列表、草稿理解、记忆检索等多个心智模型
- 页面更像“控制台能力堆叠”，而不是“围绕任务闭环的产品界面”
- 用户很难快速判断“当前最该做什么”
- 大量执行细节以块状内容暴露，普通用户难以快速理解
- 治理区、监控区与主任务流处于同一层级，复杂度暴露过早
- 当前代码组织方式更适合 Demo 或工程控制台，不适合长期扩展为复杂产品前端

## 3. 前端改造目标

前端改造建议分成三个层级目标。

### 3.1 第一层目标：任务闭环体验

让用户可以低认知成本完成以下闭环：

1. 输入任务
2. 查看系统理解与分流结果
3. 确认正式执行
4. 观察执行状态
5. 处理审批或失败恢复
6. 获取最终结果

### 3.2 第二层目标：治理与运维可视化

让高权限用户可以在不干扰普通用户主流程的前提下完成：

- actor / quota 管理
- model provider / route 管理
- tool registry 管理
- risk policy 与 change request 管理
- shadow validation / rollback / monitor 巡检

### 3.3 第三层目标：为多 Agent 与长期演进预留空间

前端需要具备未来承载以下能力的扩展基础：

- manager / specialist / reviewer 等多 Agent 协作可视化
- workflow proposal / shadow validation / rollback 的可视化管理
- Artifact 中心
- Session-first 的工作模式
- 权限感知界面裁剪

## 4. 设计原则

整体改造建议遵循以下原则：

### 4.1 主流程优先

优先展示当前用户最常用、最关键的主链路，不把所有系统能力同时暴露在首屏。

### 4.2 分层暴露复杂度

基础用户只看到任务与结果，高级用户才看到治理、监控、变更和实验能力。

### 4.3 状态驱动而不是页面驱动

用户应该优先看到“任务现在卡在哪、下一步该做什么”，而不是被迫先理解页面结构。

### 4.4 摘要优先，细节按需展开

先给可读结论，再给 trace、payload、raw JSON 和技术细节。

### 4.5 可恢复性优先

当前系统的核心特征是可审批、可恢复、可回放、可追踪，前端必须围绕“如何继续推进任务”来设计。

### 4.6 多角色兼容

viewer、operator、admin 应看到不同复杂度的操作面板，避免低权限用户面对全部高级配置项。

## 5. 建议的产品信息架构

建议把前端重构为六个一级导航域：

- 工作台
- 任务
- Sessions
- 治理
- 监控
- 设置

### 5.1 工作台

这是未来的首页，应该聚焦“今日最值得处理的事情”。

建议包含：

- 快速输入区
- 待确认草稿
- 待审批任务
- 运行中任务
- 最近失败/待恢复任务
- 最近产出结果
- 系统健康摘要

### 5.2 任务

这是围绕 task 生命周期的主操作区。

建议包含：

- 新建任务
- 任务列表
- 任务工作区
- 任务结果
- 恢复中心

### 5.3 Sessions

这是围绕上下文、记忆与历史协作链路的浏览区。

建议包含：

- Session 列表
- Session 详情
- Session health
- Reviews
- 长期记忆检索与引用建议

### 5.4 治理

这是高权限控制面，不建议与普通任务页平铺展示。

建议包含：

- Actors
- Quotas
- Tools
- Model Providers
- Model Routes
- Risk Policies
- Change Requests
- Sandbox / Shadow Validation

### 5.5 监控

这是运行巡检与故障定位入口。

建议包含：

- 运行总览
- 任务吞吐与状态分布
- 失败任务与恢复统计
- 审批阻塞统计
- Agent 运行情况
- 关键错误入口

### 5.6 设置

这是运行环境与个性化配置入口。

建议包含：

- 当前 API Base
- 当前 Actor
- 当前环境状态
- 前端实验开关
- 界面偏好

## 6. 页面级改造方案

### 6.1 首页从“功能入口页”改成“任务工作台”

当前任务面板同时堆叠了：

- 任务输入
- skill 选择
- 草稿理解
- fast path 输出
- 记忆检索
- 队列摘要
- 任务列表

建议改为三层结构：

1. 顶部状态条
2. 核心任务输入区
3. 待处理事项和近期任务

首页最重要的是回答三个问题：

- 我现在可以做什么
- 系统现在在忙什么
- 哪些任务需要我介入

### 6.2 输入区升级为“AI 任务起草器”

输入区不要再只是一个文本框加按钮，而应该包含：

- 自然语言输入
- 示例任务模板
- 最近任务复用
- 当前 session / 记忆建议
- task intent / route / deliverable spec 摘要卡片
- 正式创建任务前的确认区域

建议形成明确的三步流：

1. 输入任务
2. 查看系统理解
3. 确认执行

### 6.3 任务列表升级为“运营视图”

当前任务列表应进一步增强为：

- 状态筛选
- 关键词检索
- 最近更新时间
- 风险等级标签
- 是否需要人工处理
- 当前步骤摘要
- 快速动作入口

任务列表需要明确区分：

- 正在运行的任务
- 等待人工介入的任务
- 已完成历史任务

### 6.4 任务工作区升级为“任务驾驶舱”

每个任务页建议固定包含以下区域：

- 当前状态
- 当前卡点
- 下一步动作
- 最新产出
- 执行时间线

工作区顶部建议做成状态化 Hero 区：

- 当前任务 ID / 标题
- 状态
- 当前步骤
- session
- 是否需要审批
- 是否可恢复

任务失败或阻塞时，应直接提供主操作按钮，而不是只展示错误信息。

### 6.5 步骤和 Trace 改成时间线视图

当前步骤与 trace 适合保留技术信息，但表现形式需要升级为：

- 左侧时间线或步骤树
- 中间摘要卡片
- 右侧详情抽屉
- 原始 JSON 按需展开

优先展示：

- 这一步做了什么
- 是否成功
- 如果失败，为什么失败
- 用户接下来能做什么

### 6.6 治理页改成“风险感知控制面”

治理页不应该只是配置表单集合，而应该清楚表达：

- 当前变更目标是什么
- 风险等级是什么
- 是否需要审批
- 是否支持回滚
- 历史变更记录是什么
- 当前线上配置是否健康

### 6.7 监控页改成“巡检面板”

监控页建议聚焦异常与趋势，而不是只显示零散信息块。

优先展示：

- 任务状态分布
- 失败率趋势
- 审批阻塞数
- 模型调用健康度
- Worker / API 基础连通性
- 最近恢复动作成功率

## 7. 交互体验优化建议

### 7.1 建立全局状态条

在全局顶部持续展示：

- 当前 API Base
- 当前 Actor
- 当前环境联通状态
- 是否存在待审批或失败待恢复任务

这样可以直接降低 404、401、角色混淆等运行态问题的排查成本。

### 7.2 建立全局通知系统

建议增加统一的 toast / inline notification 体系，用于反馈：

- 输入分析成功/失败
- 任务创建成功
- 恢复动作成功/失败
- 审批操作完成
- 配置变更已生效

### 7.3 强化异步反馈

当前系统强依赖异步执行，因此需要补齐：

- skeleton loading
- 提交中的按钮状态
- 自动刷新提示
- 长任务阶段进度文案
- 任务结束后的结果提示

### 7.4 强化空态和异常态

建议统一设计以下空态与异常态：

- 没有任务
- 没有 session
- 没有审批项
- API 不可达
- 当前 actor 无权限
- 当前任务无结果
- 当前任务需要先确认草稿

## 8. 视觉系统建议

### 8.1 视觉方向

整体视觉建议从“传统后台管理页”转向“AI 运控工作台”：

- 保持克制、专业、可读
- 强调状态、结果、下一步动作
- 不追求花哨，而追求稳定感和结构感

### 8.2 布局方向

建议大量使用以下模式：

- 状态卡片
- 时间线
- 详情抽屉
- 顶部摘要条
- 右侧上下文面板

而不是继续增加纯文本块和大段无层级信息区域。

### 8.3 颜色与状态体系

建议建立统一状态色：

- running
- blocked
- failed
- completed
- pending approval
- warning

不要只靠正文文案表达状态。

### 8.4 动效策略

动效只用于提升状态变化可感知性，例如：

- 任务开始执行
- 步骤推进
- 恢复成功
- 审批通过

避免无意义动画。

## 9. 前端技术架构升级建议

## 9.1 当前技术状态

当前前端实现适合快速联调和低成本部署，但随着能力增加，已经开始暴露以下问题：

- 结构、样式、交互集中在少量超大文件中
- API 调用、页面状态、DOM 更新逻辑耦合较高
- 复用粒度低
- 页面复杂度继续增长后，维护成本会迅速上升

### 9.2 建议的升级方向

建议前端逐步升级到组件化架构。重点不是“必须换某个框架”，而是要获得以下能力：

- 组件复用
- 页面级路由
- 状态分层
- API client 封装
- 权限感知渲染
- 类型约束
- 更稳定的测试能力

### 9.3 推荐的前端分层

建议未来前端至少拆成：

- `app shell` 层
- 页面层
- 组件层
- API client 层
- 状态层
- 权限层
- 实验开关层

### 9.4 组件体系建议

建议优先抽离以下组件：

- AppShell
- StatusBar
- TaskComposer
- TaskCard
- TaskHero
- TaskTimeline
- ApprovalPanel
- TraceViewer
- SessionSummaryCard
- MemorySearchPanel
- GovernanceForm
- MonitorOverviewCard
- EmptyState
- ErrorState
- LoadingState

### 9.5 迁移策略建议

建议采用渐进迁移，而不是一次性推翻：

1. 先重构信息架构和视觉层级
2. 再抽离通用 UI 组件
3. 再封装 API client
4. 再引入更强的页面路由和状态管理
5. 最后视情况迁移到更完整的组件化框架

## 10. 权限与角色体验建议

当前系统天然具备 actor 角色边界，因此前端应该显式支持角色感知界面。

建议策略如下：

- viewer
  - 默认只看工作台、任务结果、session 浏览
  - 隐藏治理级操作入口

- operator
  - 可执行任务、处理审批、查看监控
  - 只看到必要的治理入口

- admin
  - 完整治理能力
  - 可访问模型、工具、策略、变更单与回滚入口

这能显著减少低权限用户的认知负担。

## 11. 为未来能力预留的前端设计

### 11.1 多 Agent 可视化

当前仓库已经有 Stage 5 / 6 / 7 相关数据骨架，前端需要为未来多 Agent 展示预留：

- agent 拓扑关系
- manager / specialist / reviewer 角色分工
- 子任务状态
- 评审结论
- 最终合并结果

### 11.2 Workflow Proposal 与变更验证

随着 workflow proposal、shadow validation、rollback 的推进，前端应预留：

- 提案详情
- 影响范围
- 验证结果
- 是否准入
- 回滚记录

### 11.3 Artifact 中心

结果不应只散落在任务详情中，后续建议建立独立 Artifact 视图，支持：

- Markdown
- JSON
- diff
- 审计报告
- 执行日志
- 下载与引用

### 11.4 Session-first 工作模式

当前 UI 还是明显以 task 为中心，未来可以逐步支持：

- 围绕 session 输入新任务
- 在 session 里连续推进多个任务
- 从记忆和 review 出发继续工作

## 12. 实施顺序建议

### 第一阶段：重构主流程体验

优先事项：

- 首页升级为工作台
- 输入区升级为任务起草器
- 任务工作区升级为任务驾驶舱
- 增加全局状态条与通知系统

阶段目标：

让前端从“能用的控制台”升级为“可顺畅完成任务闭环的工作台”。

### 第二阶段：建立可扩展前端骨架

优先事项：

- 抽离通用组件
- 统一 API client
- 统一加载态、空态、异常态
- 梳理导航和页面层级

阶段目标：

让前端从“单页脚本集合”升级为“可持续演进的产品前端骨架”。

### 第三阶段：完善治理与监控体验

优先事项：

- 治理页风险感知化
- 监控页巡检化
- 权限感知显示
- 关键操作回执与审计可视化

阶段目标：

让前端不仅能发起任务，还能支撑真实的运行治理。

### 第四阶段：为多 Agent 和提案系统铺路

优先事项：

- Agent 视图升级
- Workflow proposal 展示
- Artifact 中心
- Session-first 视图

阶段目标：

让前端具备承载平台长期演进的能力。

## 13. 最终结论

这次前端改造不应理解为“把静态页面改好看”，而应理解为：

把当前偏控制台式的 Web 界面，升级成一个围绕任务闭环、可恢复执行、可治理运维、可持续扩展的 AI 工作台。

建议的演进目标可以概括为三步：

1. 短期：从功能控制台升级到任务闭环工作台
2. 中期：从单页脚本界面升级到组件化产品前端
3. 长期：从任务面板升级到 AI 执行操作系统

后续如果进入实际实施阶段，建议始终优先围绕以下问题判断页面是否设计正确：

- 用户是否知道当前最该做什么
- 用户是否能一眼看到任务当前卡点
- 用户是否能低成本完成恢复、审批和继续执行
- 普通用户是否被治理复杂度过度干扰
- 前端结构是否能承载后续多 Agent 与治理扩展
